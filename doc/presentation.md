# SpotLock-Camera 📸
### オフライン写真検証システム・導入ヒアリング説明資料

**「その場所、その時間にいたこと」を、GPSや通信を使わずに証明する**

発表者: ITストラテジスト
作成日: 2026年7月2日

---

## 1. 業務背景と現在の課題

### 現場の現状
* 地下駅構内、改札、トンネル内など、**通信環境が極めて不安定またはオフライン**の現場での点検・報告業務。
* 正確な撮影場所と時間を報告書に添付する必要がある。

### 存在する不正・改ざんリスク
* **システム時刻の改ざん**: スマホの設定時刻を巻き戻して撮影する時間偽装。
* **画像の改ざん**: 撮影した写真のタイムスタンプExifデータをPCで書き換える。

<div class="alert">
💡 <b>課題:</b> 通信できない環境で、どうやって「写真と撮影時刻の正当性」を保証するか？
</div>

---

## 2. 解決策：SpotLock-Cameraの概要

### 「撮影した瞬間」に画像と時刻をセットで暗号署名
* GPSやネットワーク通信に一切依存しない、完全オフライン対応。
* カメラ内部で取得した時刻と画像データから **ECDSA (楕円曲線暗号)** で署名を作成。
* 写真内の一般には見えないメタデータ領域（`APP15`領域）に署名を埋め込む。

<div class="highlight">
<b>検証方法:</b> 提出された写真をWebの「検証ダッシュボード」にドラッグ＆ドロップするだけで、改ざんや時間偽装の有無をその場で1秒で検知。
</div>

---

## 3. システム構成（サーバー不要のゼロトラスト）

```
 [ Android アプリ ] 📸
      │  (撮影・セキュアチップ内での署名・メタデータ埋め込み)
      ▼
 [ 署名付きJPEG画像 ] (メールやチャットで通常通り送信)
      │
      ▼
 [ Web 検証ダッシュボード ] 🔍 (GitHub Pages等の静的Webサイトで動作)
      │  (ブラウザの Web Crypto API を使用したオフライン署名検証)
      ▼
 【 検証結果：改ざんなし / あり 】
```
* **サーバー維持費が「完全0円」**: 静的Webサイトのみで完結するため、データベースや検証用サーバーの維持コストが発生しません。

---

## 4. セキュリティの要：Android Keystore ＆ Google証明書

### 秘密鍵の漏洩を完全に防ぐハードウェア暗号化

1. **Android Keystore System の採用**
   * 鍵（秘密鍵）はスマホ内の安全なセキュリティチップ（TEE/StrongBox）内で生成され、**スマホの外に取り出すことは不可能**（デコンパイルされても盗まれません）。

2. **Google ハードウェア証明書 (Key Attestation)**
   * スマホのチップ自体が「本物のAndroid端末の安全なエリアで生成された鍵である」という証明書をローカルで自動発行。
   * Web側は「Googleの公開鍵」を使って、これが**本物のスマホで撮られた写真であること**をオフラインで確認可能。

---

## 5. 運用設計：自撮り写真による「初回登録」

### 誰のスマホ（鍵）かをどう特定するか？ ➔ UXを最優先した工夫

* 最初から全端末の公開鍵をWebに登録するのは、オフライン現場では面倒。
* **「初回だけ、自撮り写真（セルフィー）を送ってね」**ルールを導入。

```
 1. ユーザーが初回の「自撮り写真」を提出
      │
 2. 管理者がWeb検証画面にアップロード ➔ 「Aさん」として登録ボタンを押す
      │ (写真から公開鍵が自動で抽出され、Webに登録される)
      ▼
 以降は、その端末で撮られたすべての写真（現場写真等）が「Aさんのもの」として自動検証可能に！
```

---

## 6. 安全な鍵のローテーション（更新）設計

### 鍵の漏洩や定期更新時も、プログラム修正なしでActionsから完結

* **GitHub Variables (変数管理) の活用**
  * 歴代の公開鍵のリストと現在の鍵バージョンを、GitHubリポジトリの変数（Variables）で一元管理。
  * `{"1": "初代公開鍵", "2": "二代目公開鍵"}`

* **後方互換性の維持**
  * 鍵が新しくなっても、Web側は過去の公開鍵を保持しているため、**古いアプリで撮影した写真も引き続き検証可能**。
  * アプリの更新も、GitHub Secretsを書き換えるだけで自動ビルド。

## 7. 導入によるビジネス効果とヒアリング項目

### 期待されるビジネス効果
1. **点検報告の信頼性向上**: 写真の改ざん・時間偽装の不正を100%防止。
2. **インフラコストの最小化**: サーバー開発・維持費、通信費がゼロ。
3. **運用のしやすさ**: 現場作業員は「写真を撮って送るだけ」。

3. **初期登録プロセス**: 「初回自撮り提出」の運用フローは現場で受け入れ可能か？
